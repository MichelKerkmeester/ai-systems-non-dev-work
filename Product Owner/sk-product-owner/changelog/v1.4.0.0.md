## [**1.4.0.0**] - 2026-07-16

Packaging alignment release. It is also the release where Doc and Story became routed intents for the first time. The work landed in two commits twelve minutes apart in the same session, `Product Owner Upgrade + SKilL Refactor` then `Product Owner Update`, both timestamped 2026-07-17 in the git log even though `description.json` pins `lastUpdated` to 2026-07-16. This entry follows the declared release date and calls out the one-day gap where it matters. The skill folder renamed from `skill/` to `sk-product-owner/`, `SKILL.md` rewrote onto the sk-doc create-skill standard, and `doc-mode.md`, `doc-templates.md`, `story-mode.md` and `story-templates.md` all appeared for the first time anywhere in this repository's history. Advisor metadata, a rebuilt README, a restructured `SYNC.md` and a rebuilt Claude Project package landed alongside the rename, and a same-morning follow-up commit hardened all four content modes with decision trees, table-use guidance and new cross-mode quality checks.

> Scope: packaging, the SKILL.md rewrite and the Doc and Story debut. Task, Bug and Interactive routing proven unchanged by the 43-check regression suite before and after. Doc and Story routing is new in this release and carries no prior baseline to compare against.

---

### Skill Folder Rename And SKILL.md Rewrite

The skill folder renamed from `skill/` to `sk-product-owner/`. `SKILL.md` grew from 721 to 741 lines, version `1.1.0` to `1.4.0`, gained a keywords HTML comment, kept its seven numbered sections (`WHEN TO USE` through `INTEGRATION POINTS`) and collapsed five separate Python-fenced dict blocks down to one. Four of the five old blocks were static `INTENT_MODEL`-style dicts that nothing in a markdown file ever executed, and they folded into a single working Smart Router Pseudocode block carrying real imports, a token-boundary regex and named confidence thresholds. All 43 router fixtures pass, verified twice.

&nbsp;

### Doc And Story Mode Debut

Doc and Story joined Task, Bug and Interactive as routed intents for the first time. Neither existed anywhere under the old `skill/` tree: the tree at the commit right before the rename lists only `bug_mode.md`, `depth_framework.md`, `human_voice_rules.md`, `interactive_mode.md` and `task_mode.md` under `references/`, with no doc or story file at any earlier point in this repository's history. The new `references/doc-mode.md` (530 lines) and `assets/doc-templates.md` (729 lines), and the new `references/story-mode.md` (172 lines) and `assets/story-templates.md` (358 lines), all entered git already past `v0.100`, at `v0.105`, `v0.106` and `v0.102` respectively, evidence of drafting cycles that ran before this repository ever tracked the files. Seventeen worked examples shipped alongside them under `assets/examples/`: four Bug, six Doc, three Story and four Task. `bug-report-template.md` and `task-templates.md` carried over from the old `skill/` assets with light edits rather than shipping fresh.

### Advisor Metadata And README

`description.json` (54 lines) and `graph-metadata.json` (179 lines) landed for the first time, giving the skill a version-pinned manifest at `1.4.0.0`, a 31-entry keyword list, ten trigger examples and graph edges naming `mcp-tooling` as an enhanced skill and `sk-doc` as a sibling. The 334-line package-root `README.md`, a stale `v0.096` guide still describing `Task v0.223` and `Bug v0.127` templates several versions behind anything in the current tree, dropped to a single line: a relative symlink to the new 449-line `sk-product-owner/README.md`.

### AGENTS.md Boundary Rewrite

`AGENTS.md` rewrote its Boundaries section from a hard "you are NOT a developer, you do NOT write code" refusal into a documentation-aware boundary that allows source-backed technical HOW, architecture, APIs and explicitly labelled proposals while still forbidding fabricated implementation facts or false approval. The Strict Sequence gained a mandatory read-back step: export verification now requires Read to return non-empty content at the saved path before the response claims delivery, replacing the earlier one-line "verify the file saved successfully." A new ClickUp Handoff section lets both runtimes offer ClickUp delivery after export and forbids writing to ClickUp without explicit per-conversation approval.

`SYNC.md` rewrote from an unnumbered four-section note into an eight-section document, `OVERVIEW` through `RELATED DOCUMENTS`, on the sk-doc section standard, with frontmatter, a manual-load contract, a thirteen-row source-to-mirror inventory and a nine-point drift-check list. It states the Project Knowledge package now contains exactly thirty files, thirteen core mirrors plus seventeen worked examples.

### Kernel Drift Audit

A drift pass inside the same commit fixed four gaps in `Custom Instructions.md`. The export-path examples, previously one stale line naming only task, subtask, parent-task, bug and acceptance-criteria, split into seven per-intent lines that add doc and story. The shape list gained its fifth entry, Narrative overview, alongside Guide, Catalog, Behavior reference and Proposal. A new heading-depth rule states H1 is the title only and H2 stays a minority of the document's headings, closing what this entry calls the heading-ladder gap. The Doc deliverable's quality status became a dimensioned verdict: one pass-or-attention line each for Source safety, Shape fit, ClickUp layout, Readability and Voice.

### Mode Content Hardening (2026-07-17)

A follow-up commit the next morning, `Product Owner Update`, bumped `doc-mode.md` to `v0.106`, `doc-templates.md` to `v0.107`, `task-mode.md` to `v0.303`, `bug-mode.md` to `v0.202` and `story-mode.md` to `v0.103`. `doc-mode.md` gained a five-line decision tree for shape selection, ordered "does the reader follow an order or apply a standard" through "does the reader need orientation and story", ahead of the existing shape-by-shape prose. `doc-templates.md` added a Table Patterns subsection naming three legitimate table uses, Index, Options and trade-offs plus Conditions and outcome, and five `Section | Include when` tables, one per Doc shape, so an optional section only earns its place when its stated condition holds. Each of the other four modes gained one new quality-checklist line. Doc checks every included section against its Include when condition. Bug pairs the decisive reproduction step with its expected and actual result when the evidence supplies both. Task checks that supplied links resolve or get flagged in prose. Story adds a reader-simulation check that a developer could cut a ticket from each requirement without a follow-up question. `description.json`'s `lastUpdated` field still reads `2026-07-16T00:00:00Z`, unchanged by this pass.
